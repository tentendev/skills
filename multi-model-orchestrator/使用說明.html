Agent Skill 使用說明 · 2026-07-08

Multi-Model Orchestrator
多模型協作指揮系統

讓 Fable 5 當「技術主管」只負責規劃與整合，把深度推理交給 Opus、機械性工作交給 Sonnet、需要不同視角的問題交給 Codex——省下最貴的 Fable 用量，品質不打折。

一、功能介紹

解決什麼問題：Fable 5 是最聰明也最貴的模型。如果讓它從頭到尾自己寫 code、跑測試、改格式，大量 token 會花在「不需要頂級智力」的工作上。這套設定把 Fable 5 變成 orchestrator（指揮者）：它只做規劃、拆解、整合、最終判斷，其餘工作分派給更便宜的執行者。

團隊分工：

角色模型負責
Orchestrator（指揮）Fable 5（max reasoning）規劃、拆解任務、整合結果、最終決策
deep-reasonerOpus subagent架構設計、複雜除錯、演算法設計等重推理工作
fast-workerSonnet subagent樣板程式、測試、格式化、簡單修改等機械性工作
Codex peerOpenAI Codex CLI「不同視角」的資深工程師同儕（不是 reviewer）

殺手級機制——高風險並行盲測：遇到重大決策（架構選型、難解的 production bug），同時把同一個問題丟給 Opus 和 Codex，兩邊互相看不到對方的答案。兩者一致 → 高信心執行；兩者分歧 → 分歧點正是風險所在，由 Fable 整合出最佳解。

觸發條件：對 Claude Code 說「用 orchestrator 模式」「當 tech lead」「分派給 subagent」「讓 Opus 想、Sonnet 做」，或直接丟出任何多階段的大型開發任務（新功能、重構、複雜除錯），skill 就會啟動。

二、安裝步驟（Claude Code）

1
主模型設為 Fable 5：/model → Fable 5，/effort → max
2
把 agents/deep-reasoner.md 和 agents/fast-worker.md 複製到 ~/.claude/agents/（或專案的 .claude/agents/）
3
安裝 Codex plugin（需先裝 Codex CLI）：
/plugin marketplace add openai/codex-plugin-cc → /plugin install codex@openai-codex → /codex:setup
4
把 multi-model-orchestrator/ 資料夾複製到 ~/.claude/skills/
5
把 CLAUDE-md-snippet.md 的內容貼進專案的 CLAUDE.md

skill 負責定義「怎麼指揮」，CLAUDE.md 片段確保每個新 session 預設就用這套流程，兩者搭配使用。沒裝 Codex 也能用——skill 內建 fallback，會改用兩個獨立的 deep-reasoner 做盲測。

三、Prompt 範例（可直接複製）

範例 1：標準 tech-lead 指令（開發新功能）

PROMPT 
Goal: 幫 user dashboard 加上 CSV 匯出功能，支援目前所有篩選條件
Context: 前端在 src/pages/Dashboard.tsx，API 在 server/routes/reports.ts，
篩選邏輯已存在，不要動現有的查詢行為
You're the lead. Delegate reasoning to deep-reasoner, grunt work to
fast-worker, fresh-perspective problems to Codex. Show me your plan first,
then execute.

重點：給 Goal + Context + 授權分工，最後要求「先給計畫再動手」。

範例 2：高風險決策（並行盲測）

PROMPT 
我們的訂單服務要從單體拆出來，在「事件驅動（Kafka）」和「同步 API +
outbox pattern」之間猶豫。這是高風險架構決策：請把同樣的問題和 context
同時丟給 deep-reasoner 和 Codex，不要讓任何一方看到另一方的答案，
拿回來之後比較兩邊的結論、指出分歧點，再給我你整合後的建議。
Context 在 docs/architecture.md 和 server/services/orders/。

重點：明確說「高風險」＋「並行盲測」，觸發 skill 的 high-stakes protocol。

範例 3：複雜除錯（省 Fable 用量）

PROMPT 
Production 上偶發的 race condition：付款成功但訂單狀態卡在 pending，
log 在 logs/payment-0706.txt。用 orchestrator 模式處理——
讓 deep-reasoner 去讀 log 和相關 code 做診斷（只要回報結論，不要整份
探索過程），確定 root cause 之後把修復 patch 和 regression test 派給
fast-worker。你自己只負責把關和最終確認。

重點：明確要求「結論不要過程」，保持 orchestrator context 精簡。

範例 4：大量機械性修改

PROMPT 
把 src/ 底下所有還在用舊版 logger（console.log 和 winston 直呼）的地方
改成我們的新 wrapper（lib/logger.ts）。這是純機械性工作，直接全部派給
fast-worker 分批處理，你只要抽查結果和確認測試通過。

重點：使用者自己先判斷是機械性工作，orchestrator 只做抽查，成本最低。

四、實際 Use Case 示範

Use Case A：新功能開發（範例 1 的完整流程）

輸入

範例 1 的 prompt（CSV 匯出功能）。

產出
1
Fable 先回覆計畫：「① deep-reasoner 評估匯出策略（streaming vs 一次性）→ 驗證：回報建議＋風險 ② fast-worker 實作 API endpoint＋前端按鈕 → 驗證：既有測試全過＋新增測試 ③ 我整合並確認篩選條件正確帶入」
2
你回「go」之後，Fable 依序派工，最後回報：改了哪些檔案、deep-reasoner 的關鍵決策理由、測試結果。

效果：Fable 本體只花了規劃＋整合的 token，程式碼探索和撰寫都由 Opus/Sonnet 消化。

Use Case B：架構決策（範例 2 的完整流程）

輸入

範例 2 的 prompt（Kafka vs outbox pattern）。

產出

Fable 回報類似：「deep-reasoner 建議 outbox（理由：團隊沒有 Kafka 維運經驗、訊息量不高）；Codex 建議 Kafka（理由：未來要接三個下游系統）。分歧點在於你們對下游擴張的時程假設——如果 12 個月內確定要接多個消費端，Codex 的方向對；否則 outbox 的維運成本低得多。我的整合建議：先 outbox，但 schema 設計預留事件化空間。」

效果：兩個頂級模型的獨立判斷 + 分歧分析，比單一模型自我審查可靠得多。

Use Case C：Codex 沒裝的 fallback

輸入

同範例 2，但機器上沒有 Codex CLI。

產出

Fable 明確告知「Codex 不可用」，改開兩個互相隔離的 deep-reasoner subagent 做盲測，其餘流程不變。不會假裝有問過 Codex。

六、使用心法

  • 按認知負荷拆工，不是按檔案拆工。「需要想」的給 deep-reasoner，「答案已知、只是要做」的給 fast-worker。
  • 永遠要求結論、不要過程。orchestrator 的 context 是最貴的資源，subagent 回報越精簡越好。
  • Codex 是同儕不是 reviewer。讓它獨立解題，而不是審查別人的答案——獨立性才會暴露真正的分歧。
  • 小任務不要 orchestrate。單一明顯的修改直接做就好，分派的 overhead 要划得來。
