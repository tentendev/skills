fable-codex-setup
Fable 5 指揮官 × Codex 執行者 — 專案初始化 Skill 使用說明

一個可重複使用的 Agent Skill：一句話就把全新的 Claude Code 專案設定成「Fable 5 當大腦、Codex 與便宜模型當手腳」的省 Token 協作架構。

01這個架構在解決什麼問題？

本 Skill 的概念來自 @diegocabezas01 分享的工作流：Fable 5 很聰明但很貴，如果讓它自己讀大檔案、寫測試、跑重複性修改，昂貴的 Token 會被「粗工」消耗掉。正確的用法是把 Fable 5 當指揮官（Orchestrator）——只負責規劃、拆解任務、整合結果——然後把實際執行分派給更便宜的模型與 Codex。

原文的核心分工是：

  • Fable 5（max reasoning）＝指揮官：規劃、拆解、綜合。
  • Opus ＝ 深度推理 Sub-agent：架構設計、複雜除錯、演算法設計。
  • Sonnet ＝ 機械工作 Sub-agent：樣板程式碼、測試、格式化、簡單修改。
  • Codex ＝ 平起平坐的資深工程師：提供「不同視角」的第二意見與大量實作。重點：把它當同儕（peer），不是 reviewer。
原文還有一個高段技巧：遇到高風險決策時，讓 Opus 和 Codex 平行解同一題、互相看不到對方答案，最後由 Fable 綜合兩者最好的部分。這就是「盲測合議」，能避免兩個模型互相帶偏。

02架構總覽

Fable 5指揮官：規劃・拆解・整合（/effort max）
↓ 分派任務 ↓
deep-reasonerOpus・深度推理
fast-workerSonnet・機械執行
codex-executorHaiku 包裝 → Codex CLI
↑ 只回傳「精簡結論」 ↑

省 Token 的原理有兩層：

  1. 分工：貴的模型只做值得貴的事，粗工交給 Sonnet／Codex。
  2. 隔離：Sub-agent 的探索過程（讀檔、試錯、完整 log）都留在它自己的 context 裡，只有最終結論回到 Fable 的對話中。Fable 的 context 保持精簡，等於直接省掉最貴的那部分消耗。

其中 codex-executor 是特別為「讓 Codex 去執行」設計的：它用最便宜的 Haiku 當薄薄的調度層，透過 codex exec --full-auto 把任務整包丟給 Codex CLI 跑，跑完只驗證結果（git diff＋測試），回報摘要。Claude 這邊幾乎不花 Token。

05如何使用

步驟一：在新專案裡調用 Skill

打開任何一個新專案，對 Claude 說（或直接輸入 Skill 名稱）：

用 fable-codex-setup 把這個專案設定好

步驟二：Skill 會自動完成

  1. 環境檢查：確認 Codex CLI 與 Plugin 是否就緒，缺什麼就列出安裝指令給你。
  2. 設定大腦：在專案 .claude/settings.json 寫入 "model": "claude-fable-5"。
  3. 建立三個 Sub-agent：把範本複製到專案的 .claude/agents/（既有檔案不會被覆蓋）。
  4. 寫入 CLAUDE.md：加入「Orchestration workflow」段落，讓 Fable 每個 session 都自動知道怎麼分派工作。
  5. 回報結果：列出建立的檔案、剩下的手動步驟，並印出開工用的 Prompt。

步驟三：你手動完成兩件事

/model   → 選 Fable 5
/effort  → 選 max

06用 Prompt 驅動任務

設定完成後，用「技術主管交辦」的口吻開工（完整範本在 PROMPTS.md）：

日常開工（Tech-lead kickoff）

Goal: [你要的結果]
Context: [相關檔案、限制條件]

You're the lead. Delegate reasoning to deep-reasoner, grunt work to
fast-worker, fresh-perspective problems to Codex (codex-executor).
Show me your plan first, then execute.

高風險決策（Opus × Codex 盲測合議）

High-stakes decision: [要決定的事]

Task deep-reasoner and codex-executor on this same problem in parallel.
Do not show either one the other's answer. Then synthesize the best of
both and tell me where they disagreed.

整包丟給 Codex（最省 Token）

Hand this entirely to codex-executor: [任務描述＋明確驗收條件，
例如 done when `npm test` passes]. Only verify the result and
report back — don't implement it yourself.
關鍵心法：永遠給出「可驗證的完成條件」（done when X passes），而不是「弄到能動」。條件夠明確，Sub-agent 才能自己迭代到通過，不用一直回來問你。另外，Codex 和 Claude 之間不共享任何 context，交給 Codex 的任務描述必須完全自給自足。

07常見問題

Q：為什麼不直接讓 Fable 5 全部自己做？

A：能做，但貴。Fable 的價值在判斷力，不在打字速度。讓它讀三千行檔案、寫 20 個測試樣板，跟請首席架構師來搬磚一樣浪費。分派出去之後，粗工的 Token 由便宜模型（甚至 OpenAI 那邊的額度）吸收。

Q：Codex 和 deep-reasoner 什麼時候用哪個？

A：deep-reasoner（Opus）適合需要深入理解「這個 codebase」的推理；Codex 的價值是「不同基因的視角」——它是另一家的模型，卡關、需要第二意見、或懷疑自己想法有盲點時特別有用。真正重大的決策就兩個都問（盲測合議）。

Q：Skill 會不會弄壞我既有的設定？

A：不會。Skill 的守則是「外科手術式修改」：既有的 agent 檔案不覆蓋、settings.json 只合併不清空、CLAUDE.md 已有 Orchestration 段落就跳過，任何衝突都會先回報讓你決定。

Q：一定要裝 Codex Plugin 嗎？

A：不一定。codex-executor 直接呼叫 Codex CLI，沒有 Plugin 也能跑。但 Plugin 提供 /codex:rescue --background 等整合功能，體驗更好，建議安裝。
